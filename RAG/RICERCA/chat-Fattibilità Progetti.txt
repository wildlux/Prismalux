### Esempio di script router (Python minimale):

```python
#!/usr/bin/env python3
import subprocess
import re

DOMAINS = {
    "bio": ["animale", "cane", "lupo", "dna", "cellula", "evoluzione"],
    "religion": ["preghiera", "bibbia", "teologia", "fede", "sacro"],
    "coding": ["python", "gpu", "vram", "ollama", "modello", "training"],
    "general": []  # fallback
}

def classify_query(query: str) -> str:
    query_lower = query.lower()
    scores = {}
    for domain, keywords in DOMAINS.items():
        if domain == "general": continue
        scores[domain] = sum(1 for kw in keywords if kw in query_lower)
    return max(scores, key=scores.get) if max(scores.values()) > 0 else "general"

def run_specialized(query: str):
    domain = classify_query(query)
    model_map = {
        "bio": "qwen3.6:4b-bio",
        "religion": "qwen3.6:4b-religion", 
        "coding": "qwen3.6:4b-coding",
        "general": "qwen3.6:4b"
    }
    model = model_map[domain]
    print(f"🔍 Dominio rilevato: {domain} → modello: {model}")
    
    # Esegui con Ollama (assumes ollama già in background)
    cmd = ["ollama", "run", model, query]
    result = subprocess.run(cmd, capture_output=True, text=True)
    return result.stdout

if __name__ == "__main__":
    q = input("📝 Domanda: ")
    print("\n🤖 Risposta:")
    print(run_specialized(q))
```

## 🔁 Come funziona la distillazione per categoria

```
[Teacher Grande] (es. Qwen3.6:14B/27B, anche su cloud/CPU)
       │

[Dataset Specializzato] (es. 3.000-10.000 QA su "teologia", "biologia", "coding")
```

### 📌 Differenza chiave: Distillazione vs Fine-Tuning
| Approccio | Cosa impara | Richiede | Risultato |
|-----------|-------------|----------|-----------|
| **Fine-Tuning classico** | Risposte corrette sul dominio | Dataset umano/curato | Migliora sul tema, ma può perdere conoscenza generale |
| **Distillazione vera** | Logit + ragionamento del teacher | Teacher attivo durante training | Replica lo stile, la sicurezza e la struttura mentale del teacher |
| **SFT su output teacher** (consigliato per te) | Risposte generate dal teacher | Dataset JSONL automatico | Compromesso ottimale: qualità teacher + VRAM student |

Per il tuo hardware, la via più pratica è **SFT su dataset generato dal teacher**: il teacher produce le risposte "esperte", lo studente impara a riprodurle. È tecnicamente una distillazione semplificata, ma funziona egregiamente con QLoRA.

### Fase 2: Training dello Student ( QLoRA su 4GB VRAM )

```python
)
model = FastLanguageModel.get_peft_model(model, r=16, lora_alpha=16, target_modules=["q_proj", "v_proj"])
```

⚙️ **Accorgimenti per 4 GB VRAM:**
- `per_device_train_batch_size = 1`

| ✅ Vantaggi | ⚠️ Limiti |
|------------|-----------|
| VRAM stabile a 3-4 GB durante inferenza | Training richiede 2-8 ore (dipende da dataset/GPU) |
| Qualità vicina al teacher nel dominio scelto | Perde capacità cross-dominio (se chiedi di coding, risponde male) |
| Dataset piccolo (1k-5k esempi sufficienti) | Overfitting se il dataset è troppo omogeneo |
| Si combina perfettamente con BLHM/router | Export GGUF può richiedere ~8 GB RAM di sistema temporaneamente |

## 🔗 Come si integra con la tua architettura BLHM

La distillazione e BLHM sono **complementari**, non alternativi:

```
[Query utente]
     │
     ▼
[BLHM: matching ontologico su CPU] → Rileva: "teologia"
     │
     ▼
[Router: carica qwen3.6:4b-teologia] ← Modello distillato per categoria
     │
     ▼
[Inferenza su contesto ridotto] → Risposta esperta, VRAM < 3.5 GB
     │
     ▼
[Log + auto-aggiornamento link_w] → BLHM impara quali rami sono stati utili
```

**Sinergia**: BLHM decide *quale* modello specializzato usare, il modello distillato garantisce *qualità* in quel dominio, e il KV cache resta basso perché il contesto è già mirato.
